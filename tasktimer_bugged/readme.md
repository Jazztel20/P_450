# TaskTimer – Version projet (à tester)

## Contexte

Vous recevez une application existante nommée **TaskTimer**.  
Votre rôle est celui d’un **ingénieur QA (Quality Assurance)** chargé d’évaluer la qualité de cette application.

L’application fonctionne, mais elle contient **plusieurs anomalies intentionnelles**.

⚠️ Il ne s’agit **pas** d’un exercice de développement fonctionnel, mais d’un **projet de testing**.

---

## Objectif du projet

Votre mission consiste à :

- installer et exécuter l’application dans votre environnement de test
- analyser le comportement de l’application
- identifier les défauts, bugs et anomalies
- les décrire, les classifier et les prioriser
- définir une démarche de test cohérente
- mettre en place des tests automatisés (selon les étapes du projet)

---

## Structure du projet

```
task-timer/
├── backend/    → API REST (AdonisJS)
└── frontend/   → Interface utilisateur (Vue 3 + Vite)
```

---

## Important

- L’application **démarre correctement**
- Toutes les fonctionnalités sont **partiellement utilisables**
- Certains comportements observés sont **volontairement incorrects**
- Ces anomalies doivent être **identifiées et documentées**, pas corrigées immédiatement

---

## Consignes générales

- Ne modifiez pas le code avant les consignes de l’enseignant
- N’ajoutez pas de nouvelles fonctionnalités
- Ne “corrigez” pas les bugs avant l’étape demandée
- Basez-vous uniquement sur ce que vous observez et testez

---

## Règle essentielle

> Un test ne sert pas à prouver que l’application fonctionne,  
> mais à révéler qu’elle peut ne pas fonctionner.
